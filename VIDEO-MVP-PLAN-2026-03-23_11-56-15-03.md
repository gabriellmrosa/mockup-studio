# Video MVP Technical Plan

Written at: 2026-03-23 11:56:15 -03

## Goal

Add a first `video mode` to the current mockup editor without destabilizing the existing `photo mode`.

The MVP should allow the user to:
- choose a short animation preset;
- preview the motion in the 3D canvas;
- export the result as a short looping video;
- use the exported video as a loop in presentations, landing pages or social posts.

This plan assumes:
- `photo mode` remains the product core and current `1.0`;
- `video mode` is a later additive feature, not part of the initial `1.0` scope;
- implementation should reuse the existing canvas/export architecture as much as possible.

## MVP Scope

The MVP should support:
- short camera-based animation presets;
- export at `1920x1080`;
- duration presets such as `4s`, `6s` and `8s`;
- `30fps` default;
- optional `60fps` only if runtime remains stable;
- browser-side export using canvas capture;
- output format primarily `webm`.

The MVP should not include:
- timeline editor;
- keyframe editor;
- per-object animation authoring UI;
- audio;
- server-side rendering or transcoding;
- advanced codec/format customization;
- asset-specific cinematic presets for every device model on day one.

## Product Strategy

The first version of video should be intentionally narrow.

Recommended v1 preset set:
- `Hero Orbit`: camera orbits around the composition and returns to the initial framing.
- `Soft Push In`: subtle camera push-in with a settle near the end.
- `Parallax Sweep`: horizontal truck movement with slight orbit/parallax feeling.
- `Loop Float`: minimal camera drift designed to close seamlessly.

Reasoning:
- camera-only presets are cheaper and safer than object choreography;
- they avoid mutating scene-object transforms permanently;
- they reduce risk of device-specific animation bugs;
- they are enough to validate whether users actually need video export.

## Recommended Technical Approach

Do not use CSS keyframes for scene animation.

Preferred approach:
- animate camera and scene state inside the 3D runtime;
- record the canvas using browser APIs;
- keep animation state local to the canvas/export layer.

Core browser APIs:
- `HTMLCanvasElement.captureStream()`
- `MediaRecorder`
- `MediaRecorder.isTypeSupported()`

Preferred runtime animation approach:
- use the existing `camera-controls` integration for camera motion;
- use time-based interpolation driven by `requestAnimationFrame` or the R3F render loop;
- use easing functions and deterministic preset definitions.

## Architecture Proposal

Suggested modules:

### 1. `video-presets.ts`

Responsibilities:
- define preset ids;
- define duration defaults;
- define loopability;
- define camera animation recipe for each preset.

Suggested shape:

```ts
type VideoPresetId = "hero-orbit" | "soft-push-in" | "parallax-sweep" | "loop-float";

type VideoPresetDefinition = {
  id: VideoPresetId;
  label: string;
  defaultDurationMs: number;
  loopable: boolean;
  run: (ctx: VideoAnimationContext) => Promise<void>;
};
```

### 2. `video-runner.ts`

Responsibilities:
- run a preset against the current canvas scene;
- preserve initial camera state;
- restore camera after preview/export;
- expose lifecycle hooks such as `start`, `cancel`, `finish`.

This layer should be responsible for:
- deterministic timing;
- easing;
- loop closure behavior;
- interruption safety.

### 3. `canvas-recorder.ts`

Responsibilities:
- capture the WebGL canvas stream;
- create and manage `MediaRecorder`;
- choose MIME type via capability checks;
- assemble chunks into a downloadable `Blob`.

Suggested fallback order:
- `video/webm;codecs=vp9`
- `video/webm;codecs=vp8`
- `video/webm`

### 4. `VideoExportControls`

Responsibilities:
- select preset;
- choose duration;
- choose fps;
- start preview;
- start export;
- show recording/export state.

This should remain separate from the main photo controls to avoid bloating the current toolbar.

## Canvas Integration Strategy

The current `MockupCanvas` already handles:
- viewport controls;
- camera fitting/reset;
- PNG export;
- loading states;
- background color state localized to the canvas.

For video mode, keep the same principle:
- video preview/export state must stay local to the canvas layer;
- avoid lifting recording state to the page shell;
- avoid rerendering side panels during playback/export.

This means the likely long-term direction is:
- keep `MockupCanvas` as the orchestration boundary;
- split internals into smaller helpers/modules;
- do not turn `page.tsx` into the owner of transient animation/export state.

## Animation Design Rules

Rules for the MVP:
- presets should never permanently mutate user scene transforms;
- camera should always return to the initial saved state after preview/export;
- loopable presets must have matching start and end camera states;
- movement should be subtle and product-friendly, not cinematic for the sake of movement;
- presets must work for one or multiple scene objects.

Recommended implementation detail:
- save current camera state before preview/export;
- run preset against camera controls;
- restore saved state at the end or on cancellation.

## Export Strategy

### Format

Primary export format:
- `webm`

Reason:
- best practical browser support for canvas recording.

Possible future expansion:
- optional mp4 via post-processing or server-side pipeline;
- not recommended for the MVP if implemented purely in-browser.

### Resolution

MVP:
- `1920x1080`

Possible future options:
- `1080x1920`
- `2560x1440`
- `3840x2160`

But higher resolutions should not be part of the MVP unless performance is proven acceptable.

### Frame Rate

MVP default:
- `30fps`

Optional:
- `60fps` as advanced/high-quality option only after validation.

### Quality

Quality in the MVP should be controlled primarily by:
- resolution;
- frame rate;
- recorder bitrate.

Suggested starting bitrate ranges:
- `8-12 Mbps` for `1080p 30fps`
- `12-20 Mbps` for `1080p 60fps`

These values will require empirical tuning.

## Export Workflow Proposal

Recommended export flow:

1. User chooses preset.
2. User chooses duration/fps.
3. App disables interactive scene edits during export.
4. App resets camera to the preset start state.
5. App starts recording the canvas stream.
6. App runs the preset animation.
7. App stops recording at the exact end time.
8. App restores camera state.
9. App downloads the generated file.

Important:
- preview and export should use the same animation runner logic;
- export should avoid UI overlays being recorded if that is the desired product behavior.

## Risks And Trade-offs

### 1. Performance risk

Problem:
- real-time render + encode can be expensive.

Impact:
- frame drops;
- uneven motion;
- lower export quality than expected.

Mitigation:
- start with `1080p 30fps`;
- keep presets short;
- avoid complex per-object animation in the MVP.

### 2. Browser compatibility risk

Problem:
- codec/container support varies.

Impact:
- `mp4` may not be reliably available;
- some browsers may only support limited `webm` variants.

Mitigation:
- gate recording options via `MediaRecorder.isTypeSupported()`;
- define clear fallback order;
- keep the MVP format narrow.

### 3. Loop quality risk

Problem:
- a “loop” that visibly pops at the end will feel unfinished.

Impact:
- weak perceived quality even if technically correct.

Mitigation:
- only mark presets as loopable when the start/end state truly matches;
- prefer camera drift/orbit loops over more complex choreography.

### 4. State contamination risk

Problem:
- preview/export logic may accidentally leave the editor in a mutated state.

Impact:
- user loses expected framing or object setup.

Mitigation:
- snapshot camera state;
- restore camera after preview/export;
- keep object animation out of MVP unless isolated safely.

## Suggested Phases

### Phase 1: Internal refactor

Goal:
- prepare the canvas for transient animation/export features.

Tasks:
- isolate export logic further;
- define preset runner interfaces;
- prepare camera-state save/restore abstraction;
- ensure toolbar overlays can be excluded from capture if needed.

### Phase 2: Preview mode

Goal:
- run presets interactively without export yet.

Tasks:
- implement 2 to 4 camera presets;
- add preview controls;
- validate loop quality;
- validate compatibility across the four device models.

### Phase 3: Video export

Goal:
- record preview motion to downloadable file.

Tasks:
- implement canvas recorder;
- implement MIME fallback logic;
- export `webm`;
- add duration/fps options;
- block conflicting UI while recording.

### Phase 4: Hardening

Goal:
- turn the feature from prototype into usable product.

Tasks:
- tune bitrate and fps defaults;
- improve cancellation and cleanup behavior;
- handle unsupported browsers gracefully;
- add visual progress or recording state feedback.

## Effort Estimate

Estimated effort:

- technical spike only: `0.5-1 day`
- preview-only MVP: `2-3 days`
- export-capable MVP: `3-5 days`
- polished MVP with fallback handling and QA: `1-2 weeks`

## Recommendation

Do not implement `video mode` before `photo mode 1.0` is considered clean and stable.

When the team decides to start video:
- begin with camera-only presets;
- ship `webm`, `1080p`, `30fps`;
- treat `60fps` and larger resolutions as secondary enhancements;
- keep the feature modular so it does not bloat the current photo-first editor architecture.
